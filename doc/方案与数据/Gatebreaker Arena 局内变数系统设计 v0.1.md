# Gatebreaker Arena 局内变数系统设计 v0.1

> **版本**: v0.2 | **日期**: 2026-07-11 | **作者**: GameDesigner
> **状态**: 方案阶段 | **依赖**: 核心玩法规则 v0.3, Mode/Match 架构, 世界观与命名规范 v1.0

---

## 0. 设计前提

### 0.1 为什么需要变数

当前 Gatebreaker Arena 的对局体验由统一的规则驱动：所有玩家面对相同的球、相同的反弹、相同的弹药回复。核心碰撞和得分系统打磨得很好，但**缺少让玩家感受到「这一局不一样」的维度**。

没有变数的对称竞技最终会坍缩到最优策略重复执行——谁先掌握标准打法谁就统治所有对局。

### 0.2 设计哲学

**变数不是混乱。** 好的变数满足三条：

| 原则 | 含义 |
|---|---|
| **可预期** | 玩家在变数发生前能看到信号，可以做决策 |
| **可理解** | 效果一目了然，不需要查规则书 |
| **可反制** | 任何增益都有对应的代价或破解方式 |

### 0.3 与现有架构的关系

本文档中的所有设计遵循以下接入原则：
- **配置表驱动**：变数参数通过 `ModeRuleDefinition` / `MapRuleDefinition` 下发的配置表控制
- **热更新层实现**：所有变数逻辑放在 `App.HotUpdate.GatebreakerArena` 下
- **不破坏确定性**：Lockstep 兼容，所有随机使用伪随机（种子同步）
- **最小侵入**：能复用现有 `BallRuntimeState`、`PaddleRuntimeState`、`ServeResourceState` 字段的就复用，不轻易扩展

## 1. 变数四象限

我把「让每一局不同」拆成四个独立可叠加的维度。你可以按优先级逐个落地，每个象限独立上线都能带来可感知的变化。

```
                   主动选择（玩家决策）
                         │
           ┌─────────────┼─────────────┐
           │  象限 II    │  象限 I     │
           │  赛前 Loadout│  玩家可选   │
           │  特殊技能    │  改装机  │
           │             │             │
  赛前 ────┼─────────────┼─────────────┼─── 赛中
           │             │             │
           │  象限 III   │  象限 IV    │
           │  地图变量    │  局内事件   │
           │  地形/机关   │  道具/事件  │
           └─────────────┼─────────────┘
                         │
                   被动随机（系统生成）
```

| 象限 | 时机 | 触发方式 | 推荐优先级 |
|---|---|---|---|
| **III 地图变量** | 赛前 | 随机/轮换选图 | ⭐⭐⭐ P0 |
| **IV 局内事件** | 赛中 | 定时/条件触发投放 | ⭐⭐⭐ P0 |
| **II 赛前 Loadout** | 赛前 | 玩家主动选择装备/技能 | ⭐⭐ P1 |
| **I 改装机** | 赛前 | 玩家选择不同改装机 | ⭐ P2 |

**推荐落地顺序**: III → IV → II → I。这个顺序的原因是：III 和 IV 不要求玩家做额外决策，直接带来新鲜感；II 和 I 引入了玩家策略博弈，需等基础变数被接受后再叠加上去。

## 2. 象限 III —— 地图变量（P0 优先）

### 2.1 设计思路

现有地图是标准六边形，三个守护区均匀分布。在此基础上，通过**地图修饰器 (Map Modifier)** 在赛前随机/轮换选择一组地图特性，改变空间关系而不用新建场景。

这直接利用你现有的 `MapRuleDefinition` 扩展机制——不需要新架构。

### 2.2 地图变量清单

#### 变量 A: 墙体布局变化

| 墙体类型 | 描述 | 对局影响 |
|---|---|---|
| `WALL_NONE` | 无额外墙体（当前默认） | 基准 |
| `WALL_CENTER_PILLAR` | 场地中央一个圆形障碍物 | 阻断直线射击，制造掩体 |
| `WALL_DIVIDER` | 连接中央到边缘的隔墙，把场地一分为二 | 球需绕路，节奏变慢 |
| `WALL_CORRIDOR` | 两个平行短墙形成窄通道 | 创造"狙击线"和危险区域 |
| `WALL_ROTATING` | 场地中心一根缓慢旋转的挡杆 | 动态改变反弹路径，不可预测 |

**墙体反弹规则**:
- 墙体不影响球的归属
- 墙体反弹不触发 `SpeedGainOnPaddleHit`
- 墙体碰撞计入 `MaxCollisionIterations`
- 旋转墙体每 tick 更新角度（Lockstep 确定性推进）

**推荐首发**: `WALL_CENTER_PILLAR` + `WALL_ROTATING`——一个静态一个动态，差异明显。

#### 变量 B: 球门大小变化

| 配置 | 门宽 (HalfLength) | 门位 (Center) |
|---|---|---|
| `GOAL_STANDARD` | 3.0 | 默认六边形顶点 |
| `GOAL_WIDE` | 4.5 | 默认位置，外扩 |
| `GOAL_NARROW` | 1.8 | 默认位置，内缩 |
| `GOAL_OFFSET` | 3.0 | 沿边偏移 ±0.5 |

**设计意图**: 门宽变化直接改变攻防节奏。宽门 → 进攻强势局，窄门 → 防守拉锯局。

**平衡约束**: 同局所有玩家的门宽必须一致。门偏移可用于非对称地图（1v1 时可考虑）。

#### 变量 C: 反弹面特性

用 `MapRuleDefinition` 的字段给墙面标记不同的反弹参数：

| 反弹面类型 | MultiplySpeed | SpinEffect | 视觉标识 |
|---|---|---|---|
| `SURFACE_NORMAL` | 1.0 | 0 | 标准灰墙 |
| `SURFACE_ACCEL` | 1.3 | 0 | 红色发光边缘 |
| `SURFACE_DECEL` | 0.7 | 0 | 蓝色发光边缘 |
| `SURFACE_SPIN` | 1.0 | 0.15 | 螺旋纹路 |

**SpinEffect**: 弹后给球施加一个切线方向的速度偏移，让球的轨迹弯曲。这是低成本制造"球路不可预测"的手段——球速不变但路径弯曲，玩家必须动态调整挡板位置。

**推荐首发**: 将六边形场地的一半边设为 `SURFACE_ACCEL`，另一半设为 `SURFACE_DECEL`。玩家很快学会利用加速面进攻、减速面防守。

### 2.3 地图变量组装规则

```csharp
// 伪代码: MapRuleDefinition 扩展

{
    // --- 现有字段 ---
    public float PaddleHalfLength;
    public float GoalHalfLength;

    // --- 新增: 地图变量 ---
```

### 2.4 推荐首发组合

| 编号 | 地图名 | 墙 | 门 | 表面 | 一句话体验 |

| M4 | 加速走廊 | 两条平行短墙 | 窄门 | 通道面加速 | 高速对轰局 |

## 3. 象限 IV —— 局内事件（P0 优先）

### 3.1 设计思路

局内事件是整个方案中**让玩家最直接感受到「这一局不一样」**的维度。

核心机制：**事件投放器 (Event Spawner)**——在对局进行中，根据时间或比分条件，在场地中投放临时元素，所有玩家都能争夺或受到影响。

### 3.2 事件投放机制

**触发方式**（可叠加）：
- **定时投放**: 每 N 秒在随机位置生成一个事件道具。例如：`EventInterval = 15s`，第一局投放 4 次
- **比分触发**: 每当任意玩家得分，有概率投放。例如：`ScoreEventChance = 0.4`
- **扳机点触发**: 当比分差达到 N 时，落后方门口出现增益。例如：落后 3 分 → 门宽缩小 20%（保护落后方）

**投放逻辑**（Lockstep 兼容）：
```
1. 种子同步（从 MatchRuntime seed 派生 EventSeed）
2. 每 tick 检查投放条件
3. 满足条件 → 伪随机选择道具类型和位置
4. 位置约束：不在球门内，不在现有道具 1.5 单位内，不在墙体碰撞体内
5. 生成 EventPickup（碰撞体 + 视觉 + 时效计时器）
```

### 3.3 道具类型设计

所有道具产生后有时间限制，超时消失。同一时刻场上最多 2 个道具。

#### 类型 A: 球效果道具（直接改变场上球的状态）

| 道具 | 效果 | 持续 | 视觉 | 战术价值 |
|---|---|---|---|---|
| **加速星** | 所有己方球速 ×1.5，对手球速 ×0.8 | 8s | 黄光球尾迹 | 瞬间逆转节奏，防守方压力骤增 |
| **分裂球** | 所有己方球命中墙壁时分裂为 2 颗（每颗球速×0.8），分裂球不继承追踪分裂 | 10s / 分裂上限 3 次 | 球体闪烁 | 场面爆炸，防守难度剧增 |
| **超大球** | 所有己方球碰撞半径 ×2.5 | 6s | 球体膨胀 | 更难防守，门更易命中 |
| **幽灵球** | 所有己方球穿透墙体一次（消耗即消失），不穿透挡板 | 单次/球 | 球体半透明 | 突破墙体掩护，打对手措手不及 |
| **减速球** | 所有对手球速 ×0.5 | 5s | 蓝光球尾迹 | 防守工具，给自己喘息空间 |

**设计原则**：
- 增益道具影响「己方」或「对手」球，不区分具体是哪颗球——简单粗暴，即时理解
- 所有速度修改由 `ClampSpeed()` 重新钳制，不会超出 `MaxSpeed` / `MinSpeed`
- 道具拾取瞬间有全局音效 + 简短文字弹出（如 "SPEED UP!"）

#### 类型 B: 挡板效果道具（改变挡板能力）

| 道具 | 效果 | 持续 | 视觉 | 战术价值 |
|---|---|---|---|---|
| **长板** | 挡板长度 ×1.8 | 10s | 挡板发光延长 | 防守面积大增 |
| **磁力板** | 命中挡板的球黏附 0.3s 后沿法线弹出（玩家可微调方向） | 8s | 挡板周围粒子环 | 改变反弹时机，制造精准射击 |
| **冲刺** | 挡板移动速度 ×2.0 | 6s | 挡板尾迹 | 追球能力爆发 |
| **反弹增强** | 挡板反弹球速增益 ×2.5 | 8s | 挡板橙色闪光 | 进攻利器 |

#### 类型 C: 全局效果道具（影响场地或所有玩家）

| 道具 | 效果 | 持续 | 视觉 | 战术价值 |
|---|---|---|---|---|
| **重力场** | 场地中央出现一个缓慢移动的重力井，吸引所有球向中心偏移 | 12s | 紫色漩涡 | 改变所有球路，创造混沌 |
| **弹药补给** | 拾取者立刻恢复 2 发弹药（不突破上限） | 即时 | 弹药图标 | 战略资源，争夺激烈 |
| **冻结** | 所有对手挡板移动速度 ×0.3 | 4s | 冰霜覆盖挡板 | 最强进攻窗口 |

### 3.4 道具优先级矩阵

| 道具 | 推荐首发 | 风险 | 理由 |
|---|---|---|---|
| 长板 | ✅ | 低 | 效果直观，代码简单 |
| 加速星 | ✅ | 低 | 直接改球速，现有 `ClampSpeed` 兜底 |
| 弹药补给 | ✅ | 低 | 现有弹药系统天然支持 |
| 超大球 | 第二轮 | 中 | 需验证碰撞半径修改对 swept collision 无副作用 |
| 分裂球 | 第二轮 | 高 | 球数可能爆炸，需严格上限 |
| 重力场 | 第二轮 | 中 | 需新增球加速度系统 |
| 磁力板 | 第三轮 | 中 | 改变反弹时序，需调手感 |
| 幽灵球 | 第三轮 | 高 | 穿透逻辑需新增碰撞类型 |

### 3.5 道具拾取机制

```
拾取条件:
  - 球碰撞体与道具碰撞体重叠
  - 道具未被其他球拾取
  - 拾取球有 OwnerPlayerId（不归属无主球）

拾取后:
  - 道具销毁
  - 效果应用到拾取者（OwnerPlayerId）
  - HUD 显示效果图标 + 剩余时间
  - 全局音效播放
```

**归属规则**: 谁打的球碰到道具，效果归谁。这意味着你可以用发球去"抢"道具，也可以故意把对手的球打到道具上"送"给对方（如果球路允许）。

## 4. 象限 II —— 赛前 Loadout（P1）

### 4.1 设计思路

玩家在进入对局前，从一组预设的**装备/技能**中选择一个带入对战。类似于"带什么入场"。每个 Loadout 包含一个被动效果和一个主动技能。

这利用现有的 `PlayerRuntimeState` 扩展，不需要新的大系统。

### 4.2 Loadout 设计

#### Loadout A: 重炮手 (Artillery)

| 属性 | 内容 |
|---|---|
| 被动 | 发球初速 +30% |
| 主动 | **齐射**: 立刻向三个方向同时发球（消耗 1 弹药，生成 3 颗球）。冷却 25s |

**玩家幻想**: "我不管防守，我要用弹幕淹没对手。"

**反制**: 球多但弹药消耗快，弹药真空期长。

#### Loadout B: 守护者 (Guardian)

| 属性 | 内容 |
|---|---|
| 被动 | 挡板长度 +20% |
| 主动 | **屏障**: 在己方门前生成一个持续 3s 的能量墙，完全阻挡球进入。冷却 30s |

**玩家幻想**: "你攻不进来。"

**反制**: 进攻能力弱，依赖反击得分。屏障有冷却，时机失误代价大。

#### Loadout C: 诡术师 (Trickster)

| 属性 | 内容 |
|---|---|
| 被动 | 己方球每次墙壁反弹获得微小随机角度偏移 (±5°) |
| 主动 | **瞬移**: 挡板瞬间移动到对称位置（例如左墙→右墙对应位置）。冷却 20s |

**玩家幻想**: "你永远猜不到球会往哪飞。"

**反制**: 球的不可预测性对己方同样存在，自伤风险高。

#### Loadout D: 战术家 (Tactician)

| 属性 | 内容 |
|---|---|
| 被动 | 弹药回复速度 +50% |
| 主动 | **标记**: 标记一个对手的守护区，己方所有球在 5s 内向该区域轻微偏转。冷却 22s |

**玩家幻想**: "我的弹道计算完美。"

**反制**: 集中进攻一个目标意味着其他对手的防守压力减轻。

### 4.3 Loadout 选择流程

```
1. 进入房间 → 选择 Loadout（默认: Loadout A）
2. 可见对手选择的 Loadout（图标，不隐藏信息——这是公平竞技）
3. 倒计时结束 → 对局开始
4. 主动技能冷却从对局开始算起（部分技能有初始冷却）
```

### 4.4 Loadout 接入点

```csharp

public float ActiveAbilityMaxCooldown;
public bool HasActiveAbilityReady;

{
    public void Tick(float dt, PlayerRuntimeState[] players, BallRuntimeState[] balls, ...);
    public bool TryActivate(PlayerRuntimeState player);
}
```

## 5. 象限 I —— 改装机（P2，远期）

### 5.1 设计思路

这是四个象限中最重的一个——每个玩家选择一个改装机，改装机有独特的被动、主动技能、以及**不同的挡板/球视觉**。本质上，这是象限 II 的完全体版本。

### 5.2 为什么不现在就做

- Loadout（象限 II）已经提供了战略选择，改装机只是加了一层皮和更多技能
- 改装机需要配套的视觉资源（改装机立绘、技能特效），工作量大
- 在 Loadout 被验证后，把最受欢迎的 Loadout 升级成改装机是最经济的做法

### 5.3 以后的设计方向

| 改装机概念 | 核心机制 |
|---|---|
| 「炎核」 | 球的尾迹造成伤害效果（球路经过区域对对手有限制） |
| 「霜降」 | 挡板接触球后短暂减速球 |
| 「阻尼」 | 主动技能将球回溯到 2 秒前的位置 |
| 「工事」 | 可以在场地中放置临时障碍物 |

## 6. 组合效果与交互矩阵

当一个对局同时启用多个变数维度时，需要定义交互行为：

| 交互 | 规则 |
|---|---|
| 道具球速 × 地图加速面 | 乘法叠加，`ClampSpeed()` 最终钳制 |
| 道具长板 × Loadout 长板被动 | 加法叠加（基础 × (1 + 0.8 + 0.2) = ×2.0） |
| 分裂球 × 球速道具 | 分裂后的球继承当前球的基础速度 ×0.8，不吃额外的道具倍率 |
| 重力场 × 旋转墙 | 球同时受加速度和碰撞偏移影响，每 tick 先算加速度再算碰撞 |
| 冻结 × 冲刺 | 冲刺优先——冻结只影响基础速度，冲刺乘在冻结之后 |
| 多重球效果道具 | 后拾取的覆盖先拾取的同类型，不同类型独立计时 |

**优先级**: 负面 > 正面（确保控制效果不会被增益完全抵消）

## 7. 分阶段落地路线图

```
Phase 0（当前）: 核心碰撞 + 计分 + 发球资源 —— 已落地 ✅
│
├─ Phase 1（1-2 周）: 地图变量
│   ├── 中央障碍物 (WALL_CENTER_PILLAR)
│   ├── 旋转挡杆 (WALL_ROTATING)
│   ├── 门宽变化 (GOAL_WIDE / GOAL_NARROW)
│   └── 加速/减速墙面
│
├─ Phase 2（2-3 周）: 基础局内事件
│   ├── 事件投放器（定时 + 比分触发）
│   ├── 长板、加速星、弹药补给（3 个首发道具）
│   └── 道具拾取 + HUD 效果显示
│
├─ Phase 3（2-3 周）: 扩展局内事件
│   ├── 超大球、分裂球、重力场
│   ├── 比分落后保护机制
│   └── 道具视觉特效
│
├─ Phase 4（1-2 周）: 赛前 Loadout
│   ├── 4 个 Loadout（重炮手/守护者/诡术师/战术家）
│   ├── 主动技能系统
│   └── Loadout 选择 UI
│
└─ Phase 5（远期）: 改装机系统
    ├── 4-6 个完整改装机
    └── 改装机专属视觉
```

## 8. 平衡红线

以下是我作为设计师设定的硬性约束，任何变数都不能突破：

| 红线 | 值 | 理由 |
|---|---|---|
| 全场球数上限 | ≤ 25 | 超过后碰撞迭代成本不可控 |
| 单次速度倍率上限 | ×3.0 | 球速过快变成纯反应测试，丢失策略性 |
| 道具同时存在上限 | 2 | 超过 3 个道具在场时信息过载 |
| 负面效果最长持续 | 6s | 被控超过 6s 产生无助感 |
| 单局道具投放上限 | 8 | 避免道具喧宾夺主 |

## 9. Playtest 验证计划

每个 Phase 上线前需要验证以下问题：

### Phase 1（地图变量）验证清单
- [ ] 新墙体是否与 swept collision 正确交互？（尤其旋转墙）
- [ ] 门宽变化后，AI 的防守策略是否仍有效？
- [ ] 不同地图下对局时长是否稳定在 60-90s？
- [ ] 玩家能否在 3 秒内理解新地图的"特别之处"？

### Phase 2（局内事件）验证清单
- [ ] 道具拾取判定是否在 Lockstep 下多端一致？
- [ ] 道具效果结束时状态是否正确还原？
- [ ] 新增球数（分裂球）是否突破上限控制？
- [ ] 玩家是否能立刻说出每个道具的效果？
- [ ] 道具出现位置是否有死角（例如被墙体完全包围）？
