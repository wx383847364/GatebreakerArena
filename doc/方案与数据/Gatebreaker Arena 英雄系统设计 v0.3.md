## 0. v0.3 变更概要

### 0.1 核心变更

v0.2 建立了"改装机弱化 + 模块适配路径"框架，但模块仍然是通用池——所有改装机共享同一套模块，只按攻/守/速/乱四类匹配路径。这导致：

| 问题 | 表现 |
|---|---|
| **模块无身份感** | 同一张 STRIKE_POWER 给「霜降」和「炎核」用，感觉一样 |
| **品级无意义** | 通用模块的品级(标准/强化/精英/原型)只调数值，不带来质变 |
| **养成无方向** | 玩家不知道"我应该为哪个改装机攒什么模块" |
| **商业化空间窄** | 没有改装机专属的稀缺模块驱动付费欲望 |

v0.3 引入**专属模块系统**：

| 新增 | 说明 |
|---|---|
| **专属模块** | 改装机专属模块，绑定改装机+路径，不可跨改装机使用 |
| **品级系统** | 专属模块分精铸/调校/至高三级，品级决定适配值贡献和专属效果 |
| **适配值系统** | 替代 v0.2 的"类别模块计数"，改为适配值阈值触发里程碑 |
| **专属模块升级** | 精铸→调校→至高可升级，提供持续养成目标 |

### 0.2 与 v0.2 的兼容性

v0.2 的以下设计**保持不变**：
- 改装机基础数据削弱方案（第1节）
- 代价同步调整（第1.2节）
- 主动技能同步削弱（第1.3节）
- 路径里程碑的质性升级效果（第3节，M1/M2/M3 的具体质变内容）
- 路径类别组合设计（第2.3节）

v0.3 **替换**的内容：
- 模块系统架构（从通用池变为专属+通用混合）
- 路径触发规则（从"类别模块计数"变为"适配值阈值"）
- 改装方案构建规则（新增专属模块限制）
- 镜像模式规则（专属模块各自激活）
- 数据结构（新增专属模块定义）

## 1. 模块系统架构

### 1.1 两大模块类型

```
玩家模块收藏库
├── 专属模块池 (改装机专属)
│   ├── 绑定: 1个改装机 + 1条路径
│   ├── 品级: 精铸 / 调校 / 至高
│   ├── 数量: 10改装机 × 3路径 × 3品级 = 90张
│   └── 作用: 提供适配值 + 品级专属效果
│
└── 通用模块池 (无归属)
    ├── 类别: 攻 / 守 / 速 / 乱
    ├── 品级: 标准 / 强化 / 精英 / 原型
    ├── 数量: 24张 (v0.1已有)
    └── 作用: 提供数值增强 + 匹配路径时提供适配值
```

### 1.2 专属模块属性结构

```csharp
public class SignatureChipDefinition
{
    public string ChipId;                    // 唯一标识 (如 "SIG_FROST_DEEPFREEZE_REFINED")
    public string DisplayName;               // 显示名 (如 "寒霜之触")
    public string HeroId;                    // 绑定改装机 (如 "HERO_FROST_QUEEN")
    public string PathId;                    // 绑定路径 (如 "PATH_FROST_DEEP_FREEZE")
    public SignatureGrade Grade;             // 品级: Refined / Resonant / Supreme
    public int ResonanceValue;               // 适配值贡献: 精铸=2, 调校=4, 至高=6
    
    // 品级专属效果 (独立于里程碑)
    public ChipModifier[] GradeModifiers;    // 数值修饰器
    public QualitativeChange GradeQualitative; // 小型质变 (调校/至高才有, 可空)
    
    // 升级路径
    public string UpgradesTo;                // 升级目标模块ID (精铸→调校→至高)
    public int UpgradeCost;                  // 升级工程点成本
    
    // 视觉
    public string IconPath;                  // 图标
    public string Description;               // 描述文本
}

public enum SignatureGrade
{
    Refined,   // 精铸 - 基础品级
    Resonant,  // 调校 - 中阶品级
    Supreme    // 至高 - 顶阶品级
}
```

### 1.3 品级系统设计

| 品级 | 适配值 | 专属效果层 | 获取成本 [PLACEHOLDER] | 升级成本 [PLACEHOLDER] | 玩家感知 |
|---|---|---|---|---|---|
| **精铸** (Refined) | +2 | 数值增强 only | 200 工程点 | →调校: 300工程点+1精铸 | "走这条路的起点" |
| **调校** (Resonant) | +4 | 数值增强 + 小型质变 | 500 工程点 | →至高: 500工程点+1调校 | "已经在这条路上了" |
| **至高** (Supreme) | +6 | 高阶数值增强 + 中型质变 | 1000 工程点 | — (最高品级) | "这条路的化身" |

**品级专属效果设计原则**：
- 精铸：只提供路径方向的数值增强（如低温累积+30%），不改变行为
- 调校：数值增强 + 一个小型行为质变（如冰冻触发时附带0.5s减速）
- 至高：高阶数值增强 + 一个中型行为质变（如冰冻触发时附带1s减速+冷却过载延长）

**品级专属效果 vs 里程碑效果**：
- 品级专属效果是**模块自带**的，激活即生效
- 里程碑效果是**适配值触发**的，需要累积
- 两者**叠加生效**——至高专属激活(M3触发) = 至高专属效果 + M3里程碑质变

## 2. 路径适配值系统

### 2.1 适配值计算规则

每条路径有一个适配值计数器，激活模块按以下规则贡献适配值：

| 模块类型 | 贡献规则 | 适配值 |
|---|---|---|
| 专属模块(本路径) | 品级决定 | 精铸+2 / 调校+4 / 至高+6 |
| 专属模块(同改装机其他路径) | 不贡献 | +0 |
| 专属模块(其他改装机) | 不可装备 | N/A |
| 通用模块(类别匹配路径) | 每张贡献 | +1 |
| 通用模块(类别不匹配) | 不贡献 | +0 |

**路径类别匹配规则**（继承 v0.2）：

| 路径类别组合 | 匹配的通用模块类别 |
|---|---|
| 攻+守路径 | 攻类 或 守类 通用模块 |
| 攻+乱路径 | 攻类 或 乱类 通用模块 |
| 守+速路径 | 守类 或 速类 通用模块 |
| 攻+速路径 | 攻类 或 速类 通用模块 |
| 守+乱路径 | 守类 或 乱类 通用模块 |
| 速+乱路径 | 速类 或 乱类 通用模块 |

### 2.2 里程碑触发阈值

| 里程碑 | 适配值阈值 | 升级类型 | 玩家感知 |
|---|---|---|---|
| **M1 激活** | ≥ 2 | 数值强化（累积速度×2，时长+50%等） | "稍微强了一点" |
| **M2 质变** | ≥ 5 | **质性改变**——技能行为模式改变 | "玩法完全不同了！" |
| **M3 极化** | ≥ 9 | **终极质变**——新增子机制或完全重构 | "这已经不是原来的改装机了！" |

### 2.3 适配值计算示例

```
示例1: 「霜降」 - 极寒之路(攻+守)
  激活模块: 至高专属-绝对零度(极寒之路) + STRIKE_POWER(攻) + GUARD_LENGTH(守)
  适配值 = 6(至高专属) + 1(攻匹配) + 1(守匹配) = 8
  → 触发 M2 质变 (≥5), 未触发 M3 (<9)

示例2: 「霜降」 - 极寒之路(攻+守)
  激活模块: 调校专属-冰封领域(极寒之路) + 精铸专属-寒霜之触(极寒之路) + STRIKE_POWER(攻) + GUARD_LENGTH(守) + STRIKE_SERVE(攻) + FLOW_SPEED(速)
  适配值 = 4(调校专属) + 2(精铸专属) + 1(攻) + 1(守) + 1(攻) + 0(速不匹配) = 9
  → 触发 M3 极化 (≥9)！

示例3: 「霜降」 - 极寒之路(攻+守) — 纯通用模块
  激活模块: STRIKE_POWER(攻) + STRIKE_SERVE(攻) + GUARD_LENGTH(守) + GUARD_BOUNCE(守) + STRIKE_MULTI(攻) + GUARD_SHIELD(守)
  适配值 = 1+1+1+1+1+1 = 6
  → 触发 M2 质变 (≥5), 但需要6张全匹配才能达到
  → 对比: 1张至高专属+1张匹配通用 = 7适配值 → M2，效率高得多
```

**设计意图**: 专属模块是路径质变的**主要驱动力**——1张至高专属的适配值等于6张匹配通用模块。这保证了专属模块的不可替代性，同时通用模块仍然可以通过数量积累触发里程碑。

### 2.4 适配值与能量波动的关系

| 时间点 | 激活模块 | 适配值变化 |
|---|---|---|
| t=0s | 3 张初始激活 | 可能触发 M1(≥2) 或 M2(≥5) |
| t=20s | +1 张波动激活 | 可能升级 M1→M2 |
| t=40s | +1 张波动激活 | 可能升级 M2→M3 |
| t=55s | +1 张超载激活 | 最可能触发 M3(≥9) |

**关键设计**: 随着对局推进，激活模块越来越多，适配值累积——开局是"种子"，中盘是"生长"，终盘是"绽放"。

## 6. 通用模块与专属模块的交互规则

### 6.1 效果叠加规则

| 叠加类型 | 规则 |
|---|---|
| 专属品级效果 × 里程碑效果 | **加法叠加**——两者独立计算，同时生效 |
| 专属品级效果 × 通用模块效果 | **乘法叠加**——专属效果改改装机机制参数，通用模块改基础属性 |
| 通用模块效果 × 里程碑效果 | **加法叠加**——通用模块数值 + 里程碑数值增强 |
| 多张专属模块(同路径不同品级) | 不可同时装备——同路径只取最高品级专属模块效果 |
| 多张专属模块(不同路径) | 可同时装备——各自贡献适配值到各自路径 |

### 6.2 同路径专属模块规则

**关键规则**: 同一路径的专属模块，玩家只能装备**最高品级的一张**。

```
「霜降」 - 极寒之路:
  玩家拥有: 精铸-寒霜之触 + 调校-冰封领域 + 至高-绝对零度
  
  改装方案中只能放: 至高-绝对零度 (最高品级)
  不能同时放3张极寒之路专属模块
  
  → 鼓励玩家升级到最高品级, 而不是堆叠低品级
```

**但不同路径的专属模块可以同时装备**：
```
「霜降」改装方案示例:
  至高-绝对零度(极寒之路) + 精铸-冰凌初现(冰暴之路) + 通用模块×6
  → 有效: 极寒之路获得至高适配值, 冰暴之路获得精铸适配值
```

## 9. 镜像模式规则更新

### 9.1 专属模块在镜像模式下的处理

专属模块有改装机归属，镜像模式下双方改装机不同 → 专属模块无法镜像。解决方案：

| 模块类型 | 镜像模式处理 |
|---|---|
| **通用模块** | 镜像激活——双方激活相同的通用模块 |
| **专属模块** | 各自激活——各自从自己的专属模块中抽取 |

### 9.2 镜像模式激活流程

```
镜像模式:
  1. 双方各自选择改装机 (可能不同)
  2. 双方各自组建改装方案 (专属模块绑定各自改装机)
  3. 通用模块合并为镜像池, 专属模块各自独立
  4. 初始激活:
     a. 从镜像池抽取2张通用模块 → 双方相同
     b. 从各自专属模块中抽取1张 → 双方不同
     c. 合计3张激活 (2通用+1专属)
  5. 能量波动:
     a. t=20s: 从镜像池抽取1张通用 → 双方相同
     b. t=40s: 从各自专属模块中抽取1张 → 双方不同
     c. t=55s: 从镜像池抽取1张通用(超载) → 双方相同
     d. 合计6张激活 (4通用+2专属)
```

### 9.3 镜像模式公平性分析

| 维度 | 通用模块 | 专属模块 | 公平性 |
|---|---|---|---|
| 数值增强 | 双方完全相同 | 各自不同 | 通用公平, 专属取决于养成 |
| 路径适配值 | 双方相同(匹配相同路径) | 各自不同 | 专属适配值取决于品级 |
| 质变效果 | 双方相同 | 各自不同 | 专属质变取决于路径选择 |

**公平性保证**:
- 通用模块双方完全相同 → **数值公平**
- 专属模块各自激活，但每方只有2张专属 → **专属模块是"差异化体验"而非"强度差距"**
- 竞技差距来自**操作和策略**（选哪条路径、怎么组装），不是纯数值

**与v0.2镜像模式的差异**: v0.2中镜像模式下双方激活完全相同的模块，但因为改装机不同产生不同路径质变。v0.3中专属模块各自激活，进一步增加了差异化——但通用模块仍然镜像保证了数值公平。

```csharp
public void RecalculateResonanceValues(PlayerRuntimeState player, 
    List<UniversalChipDefinition> activeUniversal, 
    List<SignatureChipDefinition> activeSignature)
{
    var heroDef = HeroCatalog.Get(player.SelectedHeroId);
    
    // 初始化所有路径适配值为0
    foreach (var path in heroDef.ResonancePaths)
        player.PathResonanceValues[path.PathId] = 0;
    
    // 专属模块贡献适配值
    foreach (var sig in activeSignature)
    {
        if (player.PathResonanceValues.ContainsKey(sig.PathId))
            player.PathResonanceValues[sig.PathId] += sig.ResonanceValue;
    }
    
    // 通用模块贡献适配值 (按类别匹配)
    foreach (var uni in activeUniversal)
    {
        foreach (var path in heroDef.ResonancePaths)
        {
            if (path.PathCategories.Contains(uni.Category))
                player.PathResonanceValues[path.PathId] += 1;
        }
    }
    
    // 检查里程碑触发
    CheckMilestones(player);
}

public void CheckMilestones(PlayerRuntimeState player)
{
    var heroDef = HeroCatalog.Get(player.SelectedHeroId);
    
    foreach (var path in heroDef.ResonancePaths)
    {
        int rv = player.PathResonanceValues[path.PathId];
        
        int targetLevel = 0;
        if (rv >= 9) targetLevel = 3;      // M3
        else if (rv >= 5) targetLevel = 2;  // M2
        else if (rv >= 2) targetLevel = 1;  // M1
        
        int currentLevel = player.PathMilestoneLevels.GetValueOrDefault(path.PathId, 0);
        if (targetLevel > currentLevel)
        {
            TriggerMilestoneUpgrade(player, path, targetLevel);
            player.PathMilestoneLevels[path.PathId] = targetLevel;
        }
    }
}
```

| 至高专属箱 | 随机获得1张至高专属模块 | 1500工程点 (保底至高) |
| 改装机礼包 | 购买改装机时附赠该改装机全套精铸专属(3张) | 改装机解锁价格+100工程点 |
| 赛季奖励 | 段位达到指定等级赠送调校专属模块 | 免费但需竞技 |
| 专属模块升级 | 精铸→调校→至高 | 工程点+低品级模块 |

### 11.2 付费设计原则

| 原则 | 说明 |
|---|---|
| **不卖数值** | 专属模块的品级专属效果是"加速质变"而非"纯数值碾压"——精铸玩家也能通过通用模块积累触发M3 |
| **卖差异化** | 专属模块提供的是"不同的质变体验"——玩家为"我想玩这种「霜降」"付费 |
| **养成有终点** | 每个改装机90张专属模块中，玩家只需要3张至高(每路径1张)就达到了该改装机的养成上限 |
| **工程点可肝** | 所有专属模块都可以用工程点购买，工程点通过游戏免费获取 |

## 15. 风险与缓解

### 15.1 新增风险

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 90张专属模块的实现工作量极大 | 高 | 分批实现：先3改装机27张，再7改装机63张；品级专属效果用参数化实现 |
| 专属模块品级效果与里程碑效果重复/冲突 | 高 | 品级效果只做数值+小型质变，里程碑效果做大质变；两者独立计算加法叠加 |
| 镜像模式下专属模块各自激活导致不公平 | 中 | 通用模块镜像保证数值公平；专属模块每方最多2张激活，差异可控 |
| 玩家不知道该为哪个改装机攒专属模块 | 中 | 免费改装机送3张精铸专属；工程点购买可指定；改装机礼包附赠全套精铸 |
| 专属模块升级成本过高导致付费墙 | 中 | 工程点可通过游戏免费获取；精铸专属+通用模块也能触发M3(效率低但可达) |
| 同路径只取最高品级规则不直觉 | 低 | UI中明确标注"已装备更高品级，低品级不生效" |

### 15.2 确定性风险

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 适配值计算多端不同步 | 高 | 适配值纳入checksum；计算逻辑确定性 |
| 专属模块激活保底逻辑多端不一致 | 高 | 保底逻辑使用MatchSeed派生的确定性PRNG |
| 品级专属质变效果脚本多端不一致 | 高 | 质变效果用枚举+参数化实现，不用动态脚本 |

## 16. 系统交互矩阵更新

| 系统A × 系统B | 交互规则 |
|---|---|
| 专属品级效果 × 里程碑效果 | 加法叠加，独立计算 |
| 专属品级效果 × 通用模块效果 | 乘法叠加（专属改机制参数，通用改基础属性） |
| 专属模块适配值 × 通用模块适配值 | 加法叠加到同一路径 |
| 专属模块 × 改装机代价 | 品级效果可部分抵消代价（如发球速代价消除） |
| 专属模块 × 能量波动 | 波动激活新专属模块时重新计算适配值 |
| 专属模块 × 过载事件 | 专属模块不关联过载事件（过载事件仅通用模块关联） |
| 镜像模式 × 专属模块 | 通用镜像+专属各自，专属不参与镜像池 |
| 专属模块 × 地图变量 | 独立叠加，专属改改装机机制，地图改场地结构 |
| 专属模块 × 局内道具 | 独立叠加，专属是持续效果，道具是临时效果 |
