# Gatebreaker Arena 英雄签名芯片系统设计 v0.3

> **版本**: v0.3 | **日期**: 2026-07-07 | **作者**: GameDesigner
> **状态**: 方案阶段
> **依赖**: 英雄系统设计 v0.1/v0.2, 局外养成与随机触发 v0.1, 局内变数系统 v0.1, 热更新边界规范 v1
> **变更类型**: 芯片系统架构重构 — 英雄专属签名芯片 + 品级系统 + 共鸣值驱动路径质变

---

## 0. v0.3 变更概要

### 0.1 核心变更

v0.2 建立了"英雄弱化 + 芯片共鸣路径"框架，但芯片仍然是通用池——所有英雄共享同一套芯片，只按攻/守/速/乱四类匹配路径。这导致：

| 问题 | 表现 |
|---|---|
| **芯片无身份感** | 同一张 STRIKE_POWER 给冰雪女王和烈焰领主用，感觉一样 |
| **品级无意义** | 通用芯片的稀有度(Common/Rare/Epic/Legendary)只调数值，不带来质变 |
| **养成无方向** | 玩家不知道"我应该为哪个英雄攒什么芯片" |
| **商业化空间窄** | 没有英雄专属的稀缺芯片驱动付费欲望 |

v0.3 引入**签名芯片系统**：

| 新增 | 说明 |
|---|---|
| **签名芯片** | 英雄专属芯片，绑定英雄+路径，不可跨英雄使用 |
| **品级系统** | 签名芯片分精铸/共鸣/至高三级，品级决定共鸣值贡献和专属效果 |
| **共鸣值系统** | 替代 v0.2 的"类别芯片计数"，改为共鸣值阈值触发里程碑 |
| **签名芯片升级** | 精铸→共鸣→至高可升级，提供持续养成目标 |

### 0.2 与 v0.2 的兼容性

v0.2 的以下设计**保持不变**：
- 英雄基础数据削弱方案（第1节）
- 代价同步调整（第1.2节）
- 主动技能同步削弱（第1.3节）
- 路径里程碑的质性升级效果（第3节，M1/M2/M3 的具体质变内容）
- 路径类别组合设计（第2.3节）

v0.3 **替换**的内容：
- 芯片系统架构（从通用池变为签名+通用混合）
- 路径触发规则（从"类别芯片计数"变为"共鸣值阈值"）
- 牌组构建规则（新增签名芯片限制）
- 镜像模式规则（签名芯片各自觉醒）
- 数据结构（新增签名芯片定义）

## 1. 芯片系统架构

### 1.1 两大芯片类型

```
玩家芯片收藏库
├── 签名芯片池 (英雄专属)
│   ├── 绑定: 1个英雄 + 1条路径
│   ├── 品级: 精铸 / 共鸣 / 至高
│   ├── 数量: 10英雄 × 3路径 × 3品级 = 90张
│   └── 作用: 提供共鸣值 + 品级专属效果
│
└── 通用芯片池 (无归属)
    ├── 类别: 攻 / 守 / 速 / 乱
    ├── 稀有度: Common / Rare / Epic / Legendary
    ├── 数量: 24张 (v0.1已有)
    └── 作用: 提供数值增强 + 匹配路径时提供共鸣值
```

### 1.2 签名芯片属性结构

```csharp
public class SignatureChipDefinition
{
    public string ChipId;                    // 唯一标识 (如 "SIG_FROST_DEEPFREEZE_REFINED")
    public string DisplayName;               // 显示名 (如 "寒霜之触")
    public string HeroId;                    // 绑定英雄 (如 "HERO_FROST_QUEEN")
    public string PathId;                    // 绑定路径 (如 "PATH_FROST_DEEP_FREEZE")
    public SignatureGrade Grade;             // 品级: Refined / Resonant / Supreme
    public int ResonanceValue;               // 共鸣值贡献: 精铸=2, 共鸣=4, 至高=6
    
    // 品级专属效果 (独立于里程碑)
    public ChipModifier[] GradeModifiers;    // 数值修饰器
    public QualitativeChange GradeQualitative; // 小型质变 (共鸣/至高才有, 可空)
    
    // 升级路径
    public string UpgradesTo;                // 升级目标芯片ID (精铸→共鸣→至高)
    public int UpgradeCost;                  // 升级碎片成本
    
    // 视觉
    public string IconPath;                  // 图标
    public string Description;               // 描述文本
}

public enum SignatureGrade
{
    Refined,   // 精铸 - 基础品级
    Resonant,  // 共鸣 - 中阶品级
    Supreme    // 至高 - 顶阶品级
}
```

### 1.3 品级系统设计

| 品级 | 共鸣值 | 专属效果层 | 获取成本 [PLACEHOLDER] | 升级成本 [PLACEHOLDER] | 玩家感知 |
|---|---|---|---|---|---|
| **精铸** (Refined) | +2 | 数值增强 only | 200 碎片 | →共鸣: 300碎片+1精铸 | "走这条路的起点" |
| **共鸣** (Resonant) | +4 | 数值增强 + 小型质变 | 500 碎片 | →至高: 500碎片+1共鸣 | "已经在这条路上了" |
| **至高** (Supreme) | +6 | 高阶数值增强 + 中型质变 | 1000 碎片 | — (最高品级) | "这条路的化身" |

**品级专属效果设计原则**：
- 精铸：只提供路径方向的数值增强（如寒气累积+30%），不改变行为
- 共鸣：数值增强 + 一个小型行为质变（如冰冻触发时附带0.5s减速）
- 至高：高阶数值增强 + 一个中型行为质变（如冰冻触发时附带1s减速+暴风雪延长）

**品级专属效果 vs 里程碑效果**：
- 品级专属效果是**芯片自带**的，觉醒即生效
- 里程碑效果是**共鸣值触发**的，需要累积
- 两者**叠加生效**——至高签名觉醒(M3触发) = 至高专属效果 + M3里程碑质变

## 2. 路径共鸣值系统

### 2.1 共鸣值计算规则

每条路径有一个共鸣值计数器，觉醒芯片按以下规则贡献共鸣值：

| 芯片类型 | 贡献规则 | 共鸣值 |
|---|---|---|
| 签名芯片(本路径) | 品级决定 | 精铸+2 / 共鸣+4 / 至高+6 |
| 签名芯片(同英雄其他路径) | 不贡献 | +0 |
| 签名芯片(其他英雄) | 不可装备 | N/A |
| 通用芯片(类别匹配路径) | 每张贡献 | +1 |
| 通用芯片(类别不匹配) | 不贡献 | +0 |

**路径类别匹配规则**（继承 v0.2）：

| 路径类别组合 | 匹配的通用芯片类别 |
|---|---|
| 攻+守路径 | 攻类 或 守类 通用芯片 |
| 攻+乱路径 | 攻类 或 乱类 通用芯片 |
| 守+速路径 | 守类 或 速类 通用芯片 |
| 攻+速路径 | 攻类 或 速类 通用芯片 |
| 守+乱路径 | 守类 或 乱类 通用芯片 |
| 速+乱路径 | 速类 或 乱类 通用芯片 |

### 2.2 里程碑触发阈值

| 里程碑 | 共鸣值阈值 | 升级类型 | 玩家感知 |
|---|---|---|---|
| **M1 觉醒** | ≥ 2 | 数值强化（累积速度×2，时长+50%等） | "稍微强了一点" |
| **M2 质变** | ≥ 5 | **质性改变**——技能行为模式改变 | "玩法完全不同了！" |
| **M3 极化** | ≥ 9 | **终极质变**——新增子机制或完全重构 | "这已经不是原来的英雄了！" |

### 2.3 共鸣值计算示例

```
示例1: 冰雪女王 - 极寒之路(攻+守)
  觉醒芯片: 至高签名-绝对零度(极寒之路) + STRIKE_POWER(攻) + GUARD_LENGTH(守)
  共鸣值 = 6(至高签名) + 1(攻匹配) + 1(守匹配) = 8
  → 触发 M2 质变 (≥5), 未触发 M3 (<9)

示例2: 冰雪女王 - 极寒之路(攻+守)
  觉醒芯片: 共鸣签名-冰封领域(极寒之路) + 精铸签名-寒霜之触(极寒之路) + STRIKE_POWER(攻) + GUARD_LENGTH(守) + STRIKE_SERVE(攻) + FLOW_SPEED(速)
  共鸣值 = 4(共鸣签名) + 2(精铸签名) + 1(攻) + 1(守) + 1(攻) + 0(速不匹配) = 9
  → 触发 M3 极化 (≥9)！

示例3: 冰雪女王 - 极寒之路(攻+守) — 纯通用芯片
  觉醒芯片: STRIKE_POWER(攻) + STRIKE_SERVE(攻) + GUARD_LENGTH(守) + GUARD_BOUNCE(守) + STRIKE_MULTI(攻) + GUARD_SHIELD(守)
  共鸣值 = 1+1+1+1+1+1 = 6
  → 触发 M2 质变 (≥5), 但需要6张全匹配才能达到
  → 对比: 1张至高签名+1张匹配通用 = 7共鸣值 → M2，效率高得多
```

**设计意图**: 签名芯片是路径质变的**主要驱动力**——1张至高签名的共鸣值等于6张匹配通用芯片。这保证了签名芯片的不可替代性，同时通用芯片仍然可以通过数量积累触发里程碑。

### 2.4 共鸣值与共鸣波动的关系

| 时间点 | 觉醒芯片 | 共鸣值变化 |
|---|---|---|
| t=0s | 3 张初始觉醒 | 可能触发 M1(≥2) 或 M2(≥5) |
| t=20s | +1 张波动觉醒 | 可能升级 M1→M2 |
| t=40s | +1 张波动觉醒 | 可能升级 M2→M3 |
| t=55s | +1 张超载觉醒 | 最可能触发 M3(≥9) |

**关键设计**: 随着对局推进，觉醒芯片越来越多，共鸣值累积——开局是"种子"，中盘是"生长"，终盘是"绽放"。

## 6. 通用芯片与签名芯片的交互规则

### 6.1 效果叠加规则

| 叠加类型 | 规则 |
|---|---|
| 签名品级效果 × 里程碑效果 | **加法叠加**——两者独立计算，同时生效 |
| 签名品级效果 × 通用芯片效果 | **乘法叠加**——签名效果改英雄机制参数，通用芯片改基础属性 |
| 通用芯片效果 × 里程碑效果 | **加法叠加**——通用芯片数值 + 里程碑数值增强 |
| 多张签名芯片(同路径不同品级) | 不可同时装备——同路径只取最高品级签名芯片效果 |
| 多张签名芯片(不同路径) | 可同时装备——各自贡献共鸣值到各自路径 |

### 6.2 同路径签名芯片规则

**关键规则**: 同一路径的签名芯片，玩家只能装备**最高品级的一张**。

```
冰雪女王 - 极寒之路:
  玩家拥有: 精铸-寒霜之触 + 共鸣-冰封领域 + 至高-绝对零度
  
  牌组中只能放: 至高-绝对零度 (最高品级)
  不能同时放3张极寒之路签名芯片
  
  → 鼓励玩家升级到最高品级, 而不是堆叠低品级
```

**但不同路径的签名芯片可以同时装备**：
```
冰雪女王牌组示例:
  至高-绝对零度(极寒之路) + 精铸-冰凌初现(冰暴之路) + 通用芯片×6
  → 有效: 极寒之路获得至高共鸣值, 冰暴之路获得精铸共鸣值
```

## 9. 镜像模式规则更新

### 9.1 签名芯片在镜像模式下的处理

签名芯片有英雄归属，镜像模式下双方英雄不同 → 签名芯片无法镜像。解决方案：

| 芯片类型 | 镜像模式处理 |
|---|---|
| **通用芯片** | 镜像觉醒——双方觉醒相同的通用芯片 |
| **签名芯片** | 各自觉醒——各自从自己的签名芯片中抽取 |

### 9.2 镜像模式觉醒流程

```
镜像模式:
  1. 双方各自选择英雄 (可能不同)
  2. 双方各自组建牌组 (签名芯片绑定各自英雄)
  3. 通用芯片合并为镜像池, 签名芯片各自独立
  4. 初始觉醒:
     a. 从镜像池抽取2张通用芯片 → 双方相同
     b. 从各自签名芯片中抽取1张 → 双方不同
     c. 合计3张觉醒 (2通用+1签名)
  5. 共鸣波动:
     a. t=20s: 从镜像池抽取1张通用 → 双方相同
     b. t=40s: 从各自签名芯片中抽取1张 → 双方不同
     c. t=55s: 从镜像池抽取1张通用(超载) → 双方相同
     d. 合计6张觉醒 (4通用+2签名)
```

### 9.3 镜像模式公平性分析

| 维度 | 通用芯片 | 签名芯片 | 公平性 |
|---|---|---|---|
| 数值增强 | 双方完全相同 | 各自不同 | 通用公平, 签名取决于养成 |
| 路径共鸣值 | 双方相同(匹配相同路径) | 各自不同 | 签名共鸣值取决于品级 |
| 质变效果 | 双方相同 | 各自不同 | 签名质变取决于路径选择 |

**公平性保证**:
- 通用芯片双方完全相同 → **数值公平**
- 签名芯片各自觉醒，但每方只有2张签名 → **签名芯片是"差异化体验"而非"强度差距"**
- 竞技差距来自**操作和策略**（选哪条路径、怎么组牌），不是纯数值

**与v0.2镜像模式的差异**: v0.2中镜像模式下双方觉醒完全相同的芯片，但因为英雄不同产生不同路径质变。v0.3中签名芯片各自觉醒，进一步增加了差异化——但通用芯片仍然镜像保证了数值公平。

### 10.4 共鸣值计算逻辑

```csharp
public void RecalculateResonanceValues(PlayerRuntimeState player, 
    List<UniversalChipDefinition> activeUniversal, 
    List<SignatureChipDefinition> activeSignature)
{
    var heroDef = HeroCatalog.Get(player.SelectedHeroId);
    
    // 初始化所有路径共鸣值为0
    foreach (var path in heroDef.ResonancePaths)
        player.PathResonanceValues[path.PathId] = 0;
    
    // 签名芯片贡献共鸣值
    foreach (var sig in activeSignature)
    {
        if (player.PathResonanceValues.ContainsKey(sig.PathId))
            player.PathResonanceValues[sig.PathId] += sig.ResonanceValue;
    }
    
    // 通用芯片贡献共鸣值 (按类别匹配)
    foreach (var uni in activeUniversal)
    {
        foreach (var path in heroDef.ResonancePaths)
        {
            if (path.PathCategories.Contains(uni.Category))
                player.PathResonanceValues[path.PathId] += 1;
        }
    }
    
    // 检查里程碑触发
    CheckMilestones(player);
}

public void CheckMilestones(PlayerRuntimeState player)
{
    var heroDef = HeroCatalog.Get(player.SelectedHeroId);
    
    foreach (var path in heroDef.ResonancePaths)
    {
        int rv = player.PathResonanceValues[path.PathId];
        
        int targetLevel = 0;
        if (rv >= 9) targetLevel = 3;      // M3
        else if (rv >= 5) targetLevel = 2;  // M2
        else if (rv >= 2) targetLevel = 1;  // M1
        
        int currentLevel = player.PathMilestoneLevels.GetValueOrDefault(path.PathId, 0);
        if (targetLevel > currentLevel)
        {
            TriggerMilestoneUpgrade(player, path, targetLevel);
            player.PathMilestoneLevels[path.PathId] = targetLevel;
        }
    }
}
```

## 11. 商业化空间

### 11.1 签名芯片获取渠道

| 渠道 | 说明 | 成本 |
|---|---|---|
| 碎片购买 | 直接购买指定签名芯片 | 精铸200/共鸣500/至高1000 |
| 签名芯片箱 | 随机获得1张当前英雄的签名芯片 | 300碎片 (精铸70%/共鸣25%/至高5%) |
| 至高专属箱 | 随机获得1张至高签名芯片 | 1500碎片 (保底至高) |
| 英雄礼包 | 购买英雄时附赠该英雄全套精铸签名(3张) | 英雄解锁价格+100碎片 |
| 赛季奖励 | 段位达到指定等级赠送共鸣签名芯片 | 免费但需竞技 |
| 签名芯片升级 | 精铸→共鸣→至高 | 碎片+低品级芯片 |

### 11.2 付费设计原则

| 原则 | 说明 |
|---|---|
| **不卖数值** | 签名芯片的品级专属效果是"加速质变"而非"纯数值碾压"——精铸玩家也能通过通用芯片积累触发M3 |
| **卖差异化** | 签名芯片提供的是"不同的质变体验"——玩家为"我想玩这种冰雪女王"付费 |
| **养成有终点** | 每个英雄90张签名芯片中，玩家只需要3张至高(每路径1张)就达到了该英雄的养成上限 |
| **碎片可肝** | 所有签名芯片都可以用碎片购买，碎片通过游戏免费获取 |

## 15. 风险与缓解

### 15.1 新增风险

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 90张签名芯片的实现工作量极大 | 高 | 分批实现：先3英雄27张，再7英雄63张；品级专属效果用参数化实现 |
| 签名芯片品级效果与里程碑效果重复/冲突 | 高 | 品级效果只做数值+小型质变，里程碑效果做大质变；两者独立计算加法叠加 |
| 镜像模式下签名芯片各自觉醒导致不公平 | 中 | 通用芯片镜像保证数值公平；签名芯片每方最多2张觉醒，差异可控 |
| 玩家不知道该为哪个英雄攒签名芯片 | 中 | 免费英雄送3张精铸签名；碎片购买可指定；英雄礼包附赠全套精铸 |
| 签名芯片升级成本过高导致付费墙 | 中 | 碎片可通过游戏免费获取；精铸签名+通用芯片也能触发M3(效率低但可达) |
| 同路径只取最高品级规则不直觉 | 低 | UI中明确标注"已装备更高品级，低品级不生效" |

### 15.2 确定性风险

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 共鸣值计算多端不同步 | 高 | 共鸣值纳入checksum；计算逻辑确定性 |
| 签名芯片觉醒保底逻辑多端不一致 | 高 | 保底逻辑使用MatchSeed派生的确定性PRNG |
| 品级专属质变效果脚本多端不一致 | 高 | 质变效果用枚举+参数化实现，不用动态脚本 |

## 16. 系统交互矩阵更新

| 系统A × 系统B | 交互规则 |
|---|---|
| 签名品级效果 × 里程碑效果 | 加法叠加，独立计算 |
| 签名品级效果 × 通用芯片效果 | 乘法叠加（签名改机制参数，通用改基础属性） |
| 签名芯片共鸣值 × 通用芯片共鸣值 | 加法叠加到同一路径 |
| 签名芯片 × 英雄代价 | 品级效果可部分抵消代价（如发球速代价消除） |
| 签名芯片 × 共鸣波动 | 波动觉醒新签名芯片时重新计算共鸣值 |
| 签名芯片 × 量子事件 | 签名芯片不关联量子事件（量子事件仅通用芯片关联） |
| 镜像模式 × 签名芯片 | 通用镜像+签名各自，签名不参与镜像池 |
| 签名芯片 × 地图变量 | 独立叠加，签名改英雄机制，地图改场地结构 |
| 签名芯片 × 局内道具 | 独立叠加，签名是持续效果，道具是临时效果 |
