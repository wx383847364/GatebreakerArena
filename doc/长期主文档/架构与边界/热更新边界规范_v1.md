# 热更新边界规范 v1

## Summary

这份文档固定 Gatebreaker Arena 的 Unity + HybridCLR + YooAssets 分层边界。

核心目标：

- `App.AOT` 只做宿主基础设施
- `App.Shared` 只放稳定跨层契约
- `App.HotUpdate.GatebreakerArena` 承载正式玩法
- UI 只做表现和输入转发
- 正式资源统一走 YooAssets

## 分层职责

### App.AOT

允许内容：

- Unity 首场景启动器
- DI、日志、Tick、事件总线、持久化
- 平台桥接、网络底座
- YooAssets 初始化和补丁流程
- HybridCLR 热更 DLL / metadata 加载

禁止内容：

- 对局规则
- 得分判定
- 发球资源计算
- 挡板反弹公式
- AI 决策
- UI 业务流程

### App.Shared

允许内容：

- 跨 AOT / HotUpdate 的接口
- 稳定 DTO
- 跨层事件

禁止内容：

- gameplay service 实现
- `MonoBehaviour`
- `ScriptableObject`
- Editor API
- YooAssets 具体实现

### App.HotUpdate

允许内容：

- 对局状态机
- Ball / Paddle / Zone / Serve / Mode / AI 玩法逻辑
- 配表解析后的运行时 catalog
- 组合层和 presenter

约束：

- 玩法逻辑优先写成纯 C#，便于 EditMode 测试。
- Presenter 只绑定状态和转发输入，不拥有公式和结算规则。
- HotUpdate 不直接依赖 `App.AOT` 具体类型。

## 资源边界

- 正式运行时资源只通过 `IAssetsRuntime` 加载。
- 不新增 `Resources.Load` 或 `Resources.LoadAsync`。
- Editor 下允许 `YooAssetsRuntime` 通过 `AssetDatabase` 做本地加载分支。
- 第一轮迁移只验证 Editor 热更入口，Player DLL / metadata 构建在第二阶段接入。

## Gatebreaker Arena 业务拆分

- `Application`：服务注册、对局启动、组合层
- `Match`：对局状态机、计时、加时、胜负
- `Ball`：球归属、状态、离场
- `Paddle`：挡板输入与反弹
- `Zone`：守护区域判定
- `Serve`：发球弹药、冷却、许可
- `Mode`：PVE、PVP、组队模式配置
- `AI`：电脑移动和发球策略
- `UI`：HUD 绑定和反馈

## 提交前检查

- AOT 是否仍保持业务无关。
- Shared 是否只包含稳定契约。
- HotUpdate 是否没有反向引用 AOT。
- UI 是否没有写规则、公式或结算。
- 资源加载是否没有绕开 YooAssets。
